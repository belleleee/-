# Risk-HiMATE

Risk-HiMATE 是一个面向**科创企业风险识别**场景的多智能体原型系统。

它的目标不是做“聊天式问答”，而是模拟一个更接近真实审阅流程的分析系统：先从企业材料中筛出可疑证据，再让不同领域的专职智能体分别判断，之后经过反思、修正、独立核验和置信度评估，最后输出一份结构化风险报告。

这个项目适合这样理解：

- 对非技术成员：
  它是一个“自动读材料、提风险、给依据、出报告”的研究型原型。
- 对评委或老师：
  它是把 HiMATE 的多 Agent 评估机制迁移到企业风险识别任务中的一次方法改造。
- 对技术成员：
  它是一个已经可跑通的多阶段、**LangGraph-native** 风险工作流，支持 rule-based、LLM、auto fallback 三种模式，并且已经有独立置信度层、报告导出和历史趋势接口。

## 一句话理解

**输入企业文档或企业名称，系统自动识别潜在风险，输出“风险类别 + 证据 + 严重度 + 解决建议 + 置信度 + 是否需要人工复核”的结构化报告。**

## 1. 这个项目解决什么问题

现实中的科创企业，尤其是涉及 AI、数据、算法、知识产权、国际业务的企业，往往会面临很多种性质完全不同的风险：

- 算法是否存在操纵诱导、失控故障或外部滥用风险
- 数据处理是否涉及敏感信息、共享出境或存储泄漏
- 技术系统是否存在透明度不足、公平性不足或缺少可追溯机制
- 专利、技术秘密、职务发明、开源使用是否有知识产权隐患
- 出海、境外部署、出口管制、受限方名单是否构成地缘博弈风险

传统做法往往是：

- 人工读材料，耗时长
- 风险类别多，容易漏项
- 不同人判断标准不一致
- 很难把“为什么判断成这个风险”说清楚

Risk-HiMATE 试图解决的，不是“代替专业判断”，而是：

1. 帮助团队更快发现值得重点关注的风险点
2. 给出可回溯的文本证据和结构化说明
3. 提前筛出需要人工重点复核的条目
4. 为前端展示、比赛答辩和后续产品化准备统一输出格式

## 2. 这个系统是怎么工作的

整个系统可以理解成一个六步审阅流程。

从工程实现上看，当前版本已经不是单纯的手写串行 pipeline，而是按 **LangGraph-native** 的节点/状态图结构组织：

- `prepare`
- `triage`
- `domain_analysis`
- `reflection`
- `revision`
- `confidence`
- `verifier`
- `human_review`
- `finalize`
- `report`

如果本地环境里已经安装 `langgraph`，系统会直接用真正的 `StateGraph` 执行；如果当前环境还没有安装依赖，则会自动退回到兼容的顺序执行器，但节点划分和状态流保持一致。

### 第一步：接收输入

系统支持两类输入：

1. `文档输入`
   例如招股书、公告、合同、专利说明、研发报告、业务说明等。
2. `企业名称输入`
   系统先通过本地资料或外部采集器整理企业信息，再统一送入分析流程。

### 第二步：初评筛选

系统先把原始文本切成很多小段，然后快速判断：

- 哪些段落可能和风险相关
- 每段大概属于哪类风险

这一步的目标不是下最终结论，而是先把“值得看”的证据块找出来。

### 第三步：五个专职智能体分别判断

之后系统会把这些候选证据交给五个专门的风险智能体：

1. 算法安全风险智能体
2. 数据合规风险智能体
3. 科技伦理风险智能体
4. 知识产权风险智能体
5. 地缘博弈风险智能体

每个智能体都只负责自己的专业领域，不负责全包。

### 第四步：反思与修正

系统不会直接把第一轮结果当成最终答案，而是会继续检查：

- 有没有漏掉某类风险
- 有没有把风险分错类
- 严重程度是不是不合理
- 不同智能体之间对同一事实有没有明显分歧

如果发现问题，就进入修正。

### 第五步：独立核验与置信度评估

这是这个项目比较关键的地方。

系统不是只输出“有风险 / 没风险”，而是还会单独判断：

- 这条结论稳不稳
- 证据够不够强
- 多个智能体之间是否一致
- 有没有触碰隐私、合法性、伦理、公平性等治理红线

也就是说，这里有两层判断：

1. **连续置信度**
   回答“这个结果靠不靠谱”
2. **二元红线 gate**
   回答“即使结果稳定，有没有碰不能忽视的红线”

### 第六步：生成正式报告

最后，系统把结论整理成结构化风险报告，输出：

- 总体风险等级
- 总体风险分数
- 重点风险 Top3
- 每条风险的证据、建议、法规依据
- 哪些条目需要人工复核
- 置信度分解
- 风险趋势接口

## 3. 目前聚焦哪些风险

当前版本聚焦五大类风险。

### 1. 算法安全风险

- 算法操纵诱导
- 模型失控故障
- 外部对抗滥用

### 2. 数据合规风险

- 数据采集合规
- 数据存储泄漏
- 数据共享与出境

### 3. 科技伦理风险

- 全局模型可解释
- 单样本决策可追溯
- 模型迭代变更披露
- 群体信贷歧视
- 费率定价公平

### 4. 知识产权风险

- 专利侵权
- 技术秘密泄露
- 职务发明权属争议
- 开源软件合规

### 5. 地缘博弈风险

- 出口管制物项动态纳入
- 实体清单 / 受限方清单
- 外国投资审查
- 人才技术流动限制

这些定义写在 [risk_himate/app/core/taxonomy.py](/Users/belle/projects/挑战杯/risk_himate/app/core/taxonomy.py)。

## 4. 这个项目和普通“大模型问答”有什么不同

它不是一个“你问一句，我答一句”的对话机器人。

它更像一个多岗位协同的审阅流程：

- 先筛材料
- 再分工判断
- 再反思修正
- 再独立核验
- 最后统一出报告

也正因为这样，它更适合：

- 比赛展示
- 风险分析 demo
- 企业材料辅助审查
- 作为前端系统的后端分析引擎

而不适合被理解成一个通用聊天产品。

## 6. 这份报告长什么样

系统的正式输出不是一段自然语言，而是一份结构化报告。

你可以把它理解成：

- 一份给前端页面渲染的 JSON
- 一份给老师、评委或团队成员看的可读文档

报告里最值得看的内容通常有：

- `overall_risk_level`
  总体风险等级
- `overall_score`
  总体风险分数
- `top3_risks`
  最值得优先关注的三项风险
- `risk_details`
  每条风险的详细说明
- `human_review_items`
  建议人工复核的条目
- `confidence_breakdown`
  置信度是怎么来的
- `gate_flags`
  是否触碰隐私、合法性、伦理、公平性红线

报告除了按风险类别看，也可以按**生命周期阶段**看：

- 技术研发
- 数据运营
- 场景落地
- 商业出海

这让报告更适合非技术团队理解，因为它更贴近企业真实治理流程。

## 7. 置信度是怎么来的

这是当前版本的一个重要特点。

系统没有把“置信度”混在 verifier 里模糊处理，而是拆成了独立模块：

[risk_himate/app/core/confidence.py](/Users/belle/projects/挑战杯/risk_himate/app/core/confidence.py)

它会单独计算三类连续指标：

1. `signal_strength`
   看命中的风险信号强不强，证据是不是直接
2. `robustness`
   看证据是否完整、上下文是否充分、法律依据是否齐全
3. `cross_agent_consistency`
   看多个智能体对同一事实是否一致，反思修正后是否收敛

然后形成一个连续 `confidence_score`。

另外，系统还会单独判断两类红线：

- `privacy_legality_redline`
- `ethics_fairness_redline`

这两类不会直接混进平均分，而是作为治理底线单独处理。

这一层的设计参考了 TrustLLM 一类可信 AI 评估框架，但不是原样照搬，而是根据“多智能体风险识别”任务做了重组。

## 9. 当前还不够完善的地方

这个项目已经能用来演示，但它仍然是一个**研究型原型**，不是生产级产品。

目前最明确的待优化点包括：

### 1. Prompt 设计还需要继续收紧

- 各 agent 的边界还可以更清晰
- evidence、subtype、severity 的一致性还可以更强
- 某些场景下仍然可能出现“为了输出而输出”

### 2. 地缘博弈风险仍然偏容易误报

尤其是：

- “国际合作”
- “合作不确定性”
- “出海计划”

这类表述如果没有更硬的监管信号，理论上不该轻易判成地缘博弈风险。

### 3. PDF 链路还不够产品化

当前能分析 PDF，但仍然是：

1. 先抽文本
2. 再把文本送入 pipeline

它还没有原生 `--pdf` 参数，也还没有 OCR、页码回溯这些能力。

### 4. 外部数据采集仍然是原型状态

现在已经有本地采集器和外部采集骨架，但还没到完整商用接入程度。

### 5. 置信度还需要真实样本标定

虽然已经有独立置信度层，但当前权重仍然是启发式的，后续需要用更多真实案例做标定。

### 6. 前端展示层还可以更强

比如：

- 更直观的阶段可视化
- 风险评分仪表盘
- 示例输入与输出对照
- 更完整的答辩展示页

## 10. 仓库结构怎么理解

如果你不是技术成员，可以这样理解仓库：

### `HiMATE-main/`

这是论文原始方法的参考实现，不是你们现在比赛要直接展示的主系统。

### `risk_himate/`

这是当前真正的项目主体，里面放的是“企业风险识别版”的实现。

### `risk_himate/app/agents/`

这里相当于“系统里的不同岗位”：

- 初评员
- 五类专职分析员
- 反思员
- 修正员
- 核验员

### `risk_himate/app/core/`

这里是系统的基础规则：

- 风险分类
- 数据结构
- 评分逻辑
- 置信度逻辑
- 报告生成逻辑

### `risk_himate/app/llm/prompts/`

这里是每个智能体实际使用的提示词模板。

### `risk_himate/app/workflows/pipeline.py`

这里是整个系统对外暴露的主调度入口，它会调用底层的 LangGraph-native 工作流来执行完整流程。

### `risk_himate/app/workflows/state_graph.py`

这里定义了当前真正的状态图工作流。也就是说，系统已经具备明确的节点、边和分支，而不是把所有阶段硬编码在一个长函数里。

### `tests/`

这里是自动化测试，确保系统修改后不容易出错。

## 12. 如果你是技术成员，怎么启动它

### 安装依赖

```bash
python3 -m pip install -r requirements.txt
```

### 直接分析一段文本

```bash
python3 -m risk_himate.app.main \
  --input-type document \
  --company "测试企业" \
  --text "公司计划将用户行为数据存储在境外云平台，并用于智能推荐，尚未明确披露用户授权机制。" \
  --llm-mode auto
```

### 使用本地 LM Studio

```bash
LLM_BASE_URL=http://127.0.0.1:1234/v1 \
LLM_API_KEY=lm-studio \
LLM_MODEL=qwen2.5-7b-instruct-mlx \
python3 -m risk_himate.app.main \
  --input-type document \
  --company "测试企业" \
  --text "本公司收集用户面部识别数据用于广告推送，数据存储在境外服务器，未向用户明确告知。" \
  --llm-mode llm
```

### 只导出正式 JSON 报告

```bash
python3 -m risk_himate.app.main \
  --input-type document \
  --company "测试企业" \
  --text "这里放分析文本" \
  --llm-mode auto \
  --report-only \
  --output risk_himate/output/my_report.json
```

### 导出可读文档

```bash
python3 -m risk_himate.app.main \
  --input-type document \
  --company "测试企业" \
  --text "这里放分析文本" \
  --llm-mode auto \
  --report-only \
  --output risk_himate/output/my_report.json \
  --rendered-report risk_himate/output/my_report.docx
```

支持的可读导出格式包括：

- `.md`
- `.html`
- `.rtf`
- `.docx`

## 13. 一份结果怎么读

如果你第一次看系统输出，优先关注这几项：

### 先看总览

- `overall_risk_level`
- `overall_score`
- `top3_risks`

### 再看每条风险

重点字段是：

- `category`
- `subtype`
- `severity`
- `evidence`
- `suggestion`
- `legal_basis`

### 最后看是否需要人工介入

- `human_review_items`
- `confidence_breakdown`
- `gate_flags`

如果一条风险既低置信度，又触发红线，那它通常就是最应该优先人工复核的对象。
